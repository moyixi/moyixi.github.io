基数排序

发表于 2020-05-10 | 分类于 排序算法

从低位到高位逐渐对数字进行排序。

桶排序

发表于 2020-05-10 | 分类于 排序算法

桶排序可看做计数排序的升级版，将数据分到有限的桶里，每个桶再分别排序。
先设定一个定量的数组当做空桶，遍历数据，将数据放入一个个对应的桶中，对每个不是空的桶进行排序，并从不是空的桶里把排好序的数据拼接起来。

计数排序

发表于 2020-05-10 | 分类于 排序算法

计数排序不是基于比较的排序算法,是桶排序思想中的一种，可看做每个桶的大小为1。需要额外开辟空间,空间复杂度O(n+k),时间复杂度O(n+k)，待排序的数据值存储在额外开辟的数组空间中，计数排序要求数据必须是有确定范围的整数，常用于数量大但是范围小的排序，例如人员年龄排序，高考名次排名。
1、先找出待排序数组中最大和最小的元素。
2、统计数组中每个值为i的元素出现的次数，存入数组的第i项.
3、将数组中的每个元素按顺序放入新数组中。

堆排序

发表于 2020-05-10 | 分类于 排序算法

堆是一种完全二叉树，且满足父节点大于孩子结点
完全二叉树：节点顺序满足从上往下，从左往右。
堆排序过程：
1、先构造一个大顶堆，取堆顶数字(即最大值)
2、将剩下的数字构建一个大顶堆，取堆顶数字(即剩下值中的最大值)
3、重复以上操作，直到取完堆中的数字，得到一个从大到小排列的序列。

堆的构建过程(heapify)

从下往上的去构建堆，先在最底层的节点中，找出父节点与两个子节点中的最大值，和父节点做交换。
用数组去表示一棵完全二叉树，按层序，从上至下，从左往右的表示。则可知
对于i节点，其
父节点parent = (i-1)/2
左孩子c1 = 2 * i+1
右孩子c2 = 2 * i+2
当一个大顶堆的根结点发生变化时，通过heapfiy操作来将其重新变为大顶堆。
而想要把一个乱序的数组变为大顶堆，build需要从底往上的进行heapfiy操作。即从最后一个结点的parent节点开始，到根节点为止不断地heapify。
想要把已经构建好的大顶堆变成有序数组，需要把堆顶元素和末尾元素交换，再令堆的大小自减1，并继续做heapfiy操作，直到整个数组交换完毕为止。

快速排序

发表于 2020-05-10 | 分类于 排序算法

快排的思想为，首先找到一个基准，将大于基准的放在右边，小于基准的放在左边。然后再对基准左右的两个子数组进行快排。

partition过程

可以看出，快排中最为重要的就是partition的过程。现在对上面的partition过程进行分析。
先取待排序数组的最末尾位置为pivot，用l当做左指针从左往右寻找，r当做右指针从右往左寻找。
当l寻找到第一个大于pivot的元素，r寻找到第一个不小于privot的元素时暂停本次寻找。
由于最终目的是把所有小于pivot的元素放在左边，大于pivot的元素放在右边，故对上一步找到的那两个元素进行交换。
l和r这两个指针对数组的扫描直到l>r时停止，这代表着整个数组已经按照pivot划分为两部分了。
上述循环结束之后，l指针左边的元素都小于等于pivot，r右边的元素都大于pivot，故l指针当前指向的元素大于pivot，而r指针当前指向的元素小于等于pivot，而此时pivot还在数组的最末端，未参与上述交换过程，为了将pivot放入正确的位置，可以将其与当前l指针指向的元素交换。交换后pivot位于l指针所在位置，可将其作为返回值，用于pivot左右子数组的快排分界线。

第二种partition思路
同样选定待排序数组的最末尾位置为pivot，目的是在遍历一遍数组的过程中，把比pivot小的数字从左往右的放在数组中。
故使用location记录下一个要放的位置，每次找到一个比pivot小的元素，将其与location位置的数字交换，并将location自增1。
最后同样的要把数组最末尾位置与location交换，并返回location用于左右子数组的快排。

pivot的选择

上述partition过程中都是选择的最末尾位置为pivot，这可能使得比较极端的pivot使左右子数组长度不均衡，一种简单的优化方式是取最左，最右，中间三个值的中位数当做pivot。

归并排序

发表于 2020-05-10 | 分类于 排序算法

将长度为n的数组划分位两个长度为n/2的子数组，再分别对子数组采用归并排序，将两个排序好的子数组合并成一个,归并排序时间复杂度为O(nlogn),缺点是需要开辟额外的空间，空间复杂度为O(n)。

选择排序插入排序希尔排序

发表于 2020-05-09 | 分类于 排序算法

选择排序与插入排序是最为直观的两种排序，平均时间复杂度均为O(n2)，空间复杂度为O(1)。

选择排序

首先在数组中选择最小的元素，放入数组的起始位置，然后再在剩余未排序元素中继续寻找最小的元素，放入已排序数组的末尾。
故要嵌套两层循环，外层循环代表要选择过程的次数，内层循环从未排序元素中找到最小的那个。

插入排序

插入排序类似于抓扑克牌的过程，每次取到一张牌时，从前往后比较，将其放在合适的位置。
视第0个元素已排序，从第一个元素开始，取出一个元素(cur = arr[i])， 在已排序元素中从后往前依次对比，若已排序元素更大，则将其往后挪一位，直到已排序元素不大于被取出的元素时，将被取出的cur放入。

希尔排序

希尔排序是插入排序的改进版，
第一步、分组，找到一个间隔(间隔一般为总长度的一半)，按间隔把待排序元素分为一组。
第二步、在组内使用插入排序法进行排序。
第三步、重新设置间隔(在原来间隔的基础上减半)，重新分组，在组内再排序。
第四步、不管数组有多大，最终分组间隔都会变成1，完成最后一轮排序后整个数组有序。

在普通插入排序中，一个靠后的数字，要想排到前面，需要和他前方的所有数字做一次比对，而希尔排序中，若间隔比较大，需要比对的次数减少，若间隔比较小，需要比对的数据组内距离比较小，同样的减少了比对次数，故效率有所提高。但由于希尔排序跳着排，故不稳定。

希尔排序的改进

可以通过对希尔排序间隔序列的优化而改进希尔排序
采用Knuth序列，效率会更高。

冒泡排序及其优化

发表于 2020-05-09 | 分类于 排序算法

冒泡排序(Bubble Sort)

依次比较相邻元素，把较大的交换到右端，走完一趟后，最大值放到数组最末端，放完后第二趟不需要再和最大值比较，故每趟比较都比上一趟比较少一次。
冒泡排序用了两层循环嵌套，外层循环控制排序趟数，内层循环控制每一趟排序多少次，故时间复杂度为O(n2)，空间复杂度为O(1)。

可以看出同样的数据情况下，经过优化后的冒泡排序比原来的冒泡排序减少了两趟

冒泡排序的优化二

第一种优化方法，因为记录了某一趟是否发生了交换，不发生交换则说明整体有序。
第二种优化方法，可以记录下每一趟过程中最后的交换位置，最后交换位置代表着此位置之后没有经历过交换了，即此位置之后的所有元素都在其本位，下一趟可以不用与之比较。
可以看出，之前两种冒泡排序的内层循环结束条件为 j < len - 1 - i;即每走一趟(i++) 下一趟就少进行一次比对，因为每趟必有一个元素回归本位，但每趟回归本位的元素可能不止一个，可以用lastchange记录每趟比对最后一次交换发生的位置，在下一趟时比对到当前位置即可。即每次更新内层循环的结束条件。

冒泡排序的优化三

第三种优化方法可以称为双向冒泡排序，即每次除了从左往右的将最大值冒泡至最后位置外，还从右往左的将最小值冒泡至最前位置，同时保持着优化二的优点————记录每趟比对后最后一次发生交换的位置。则，双向冒泡需要记录两个位置，将数组分为左中右三块，左边所有元素在将小值冒泡至左边时有序，右边所有元素在将大值冒泡至右边时有序，中间所有元素是待冒泡的元素，所以要用两个变量leftlast和rightlast记录中间区间的左右两端。leftlast代表向左冒泡时最后一个发生交换的位置，rightlast代表向右冒泡时最后一个发生交换的位置。

数据库索引

发表于 2020-05-09 | 分类于 MySQL

索引是帮助MySQL高效获取数据的排好序的数据结构
索引数据结构：

  • 二叉查找树
  • 红黑树
  • Hash表
  • B-Tree

MySQL索引一般不用二叉查找树
因为若所插入的数据是单边增长的数据，则二叉查找树演变成链表，查找效率极低。

红黑树
当插入数据左右失衡时，会自动调整的一种二叉树
缺点：数据库中存在着动辄百万千万的大量数据，此时红黑树的高度会变得越来越大，在查找叶子结点时，所耗费的时间等于红黑树的高度，效率降低。

Hash表
插入时，把索引的每一个字段做一个Hash运算，将得到的散列值和索引所在的磁盘文件地址指针存入一张Hash映射表中。
查找时，对所要查找的元素做一个Hash运算，用得到的散列值在映射表里定位到对应的磁盘文件地址指针，精确查找时速度很快。
缺点：无法做到范围查找，且若有大量重复键值的情况，Hash索引效率会由于Hash碰撞问题而变得很低。

B-Tree
给二叉查找树的节点一次性分配更多的容量，让一个节点里存更多的索引元素。
叶节点具有相同的深度，叶节点的指针为空，所有索引元素不重复，节点中的数据索引从左到右递增排列

缺点：范围查找也很麻烦。

B+Tree(B-Tree变种)
非叶子结点不存储data，只存储索引，可以放更多的索引（冗余）。叶子节点包含所有的索引字段。叶子节点用指针连接，提高区间访问的性能

B+Tree把所有的索引元素都存在叶子节点中，然后把出于中间位置的索引元素提上去，放到非叶子节点，作为冗余存放。
通过下面语句，可知MySQL一个节点的默认大小为16384(16KB)

优势：
B-Tree的非叶子节点存储data，而B+Tree的非叶子节点不存储data，这样B+Tree就可以存入更多的索引。Hash表无法做到范围查找，B-Tree范围查找也很麻烦，而B+Tree在做范围查找时由于叶子节点从左到右依次递增且其间有指针连接，故非常方便。

MyISAM存储引擎索引实现
MyISAM索引文件和数据文件是分离的(非聚集)

非聚集索引先查找到元素，和与之存储在一起的磁盘文件地址指针，再通过磁盘文件地址指针找到目标。

InnoDB索引实现(聚集)

表数据文件本身就是按B+Tree组织的一个索引结构文件
聚集(聚簇)索引-叶节点包含了完整的数据记录

InnoDB的表必须要有主键，并且推荐使用整型的自增主键。原因：整型的主键相互比较时比较快，且占用空间小。若不用自增的主键，出现非递增的情况时，有较大的维护成本。

联合索引的底层存储结构 ：

把几个元素放入一个节点中存储，排序规则按字段逐个比较。
联合索引的最左前缀匹配原则即where条件必须有联合索引的第一个字段(联合索引中最左边的索引)，与where条件的顺序无关。因为联合索引的数据存储时，按照第一个字段来进行排序，只有当第一个字段相等，才按照第二个字段排序。所以在整个数据结构中，除第一个字段外都是全局无序，局部有序状态，无法按索引找到。当遇到范围查询(>、<、between、like)就会停止匹配，因为当第一个字段为范围值而第二个字段为确定值时，在第一个字段所查找的范围内，第二个字段不是有序的，无法使用索引。
