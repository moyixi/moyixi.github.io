数据库索引

发表于 2020-05-09 | 分类于 MySQL

索引是帮助MySQL高效获取数据的排好序的数据结构
索引数据结构：

  • 二叉查找树
  • 红黑树
  • Hash表
  • B-Tree

MySQL索引一般不用二叉查找树
因为若所插入的数据是单边增长的数据，则二叉查找树演变成链表，查找效率极低。

红黑树
当插入数据左右失衡时，会自动调整的一种二叉树
缺点：数据库中存在着动辄百万千万的大量数据，此时红黑树的高度会变得越来越大，在查找叶子结点时，所耗费的时间等于红黑树的高度，效率降低。

Hash表
插入时，把索引的每一个字段做一个Hash运算，将得到的散列值和索引所在的磁盘文件地址指针存入一张Hash映射表中。
查找时，对所要查找的元素做一个Hash运算，用得到的散列值在映射表里定位到对应的磁盘文件地址指针，精确查找时速度很快。
缺点：无法做到范围查找，且若有大量重复键值的情况，Hash索引效率会由于Hash碰撞问题而变得很低。

B-Tree
给二叉查找树的节点一次性分配更多的容量，让一个节点里存更多的索引元素。
叶节点具有相同的深度，叶节点的指针为空，所有索引元素不重复，节点中的数据索引从左到右递增排列

缺点：范围查找也很麻烦。

B+Tree(B-Tree变种)
非叶子结点不存储data，只存储索引，可以放更多的索引（冗余）。叶子节点包含所有的索引字段。叶子节点用指针连接，提高区间访问的性能

B+Tree把所有的索引元素都存在叶子节点中，然后把出于中间位置的索引元素提上去，放到非叶子节点，作为冗余存放。

优势：
B-Tree的非叶子节点存储data，而B+Tree的非叶子节点不存储data，这样B+Tree就可以存入更多的索引。Hash表无法做到范围查找，B-Tree范围查找也很麻烦，而B+Tree在做范围查找时由于叶子节点从左到右依次递增且其间有指针连接，故非常方便。

MyISAM存储引擎索引实现
MyISAM索引文件和数据文件是分离的(非聚集)

非聚集索引先查找到元素，和与之存储在一起的磁盘文件地址指针，再通过磁盘文件地址指针找到目标。

InnoDB索引实现(聚集)

表数据文件本身就是按B+Tree组织的一个索引结构文件
聚集(聚簇)索引-叶节点包含了完整的数据记录

InnoDB的表必须要有主键，并且推荐使用整型的自增主键。原因：整型的主键相互比较时比较快，且占用空间小。若不用自增的主键，出现非递增的情况时，有较大的维护成本。

联合索引的底层存储结构 ：

把几个元素放入一个节点中存储，排序规则按字段逐个比较。
联合索引的最左前缀匹配原则即where条件必须有联合索引的第一个字段(联合索引中最左边的索引)，与where条件的顺序无关。因为联合索引的数据存储时，按照第一个字段来进行排序，只有当第一个字段相等，才按照第二个字段排序。所以在整个数据结构中，除第一个字段外都是全局无序，局部有序状态，无法按索引找到。当遇到范围查询(>、<、between、like)就会停止匹配，因为当第一个字段为范围值而第二个字段为确定值时，在第一个字段所查找的范围内，第二个字段不是有序的，无法使用索引。

DNS寻址过程

发表于 2020-05-08 | 分类于 计算机网络

计算机在网络中进行通讯只能识别IP地址，而无法识别域名，在访问网站时，在浏览器中输入域名，然后DNS(Domain Name System)服务器将域名翻译为IP地址，并调出IP地址所对应的网页。
由于互联网上有大量主机，单一的DNS服务器无法解决单点故障、无法保证通信容量以及无法临近所有的查询主机和维护困难等问题，所以DNS不适合使用单点的集中式设计，而是使用分布式集群的工作方式，通过使用分布式层次数据库和缓存方法来解决以上问题。

根DNS服务器：因特网上有13个根DNS服务器，大部分分布在北美洲，可以将这13个根DNS服务器视为单个的服务器，但是每台服务器实际上是一个冗余的计算机网络以提供安全性和可靠性；
顶级域DNS服务器：负责顶级域名，如com，org，net，edu，gov以及各个国家的顶级域名的转换。
权威DNS服务器：因特网上，具有公共可访问主机的每个组织机构必须公共可访问的DNS记录，这些记录将主机名映射为IP地址。一个组织的权威DNS服务器收藏了这些DNS记录，多数大学和大公司实现和维护它们自己的基本和辅助（备份）权威DNS服务器；当然，也可以通过付费的方式，将相关的信息插入到其它权威服务器中；
本地DNS服务器：本地DNS服务器通常邻近其所在网络的其他主机。当主机发出DNS请求时，该请求被发往本地DNS服务器，它起着代理的作用，并将请求转发到DNS服务器层次结构中。
DNS缓存：每当DNS服务器发出请求后收到回答时，就将回答的内容缓存在它自己的主机空间上。这样，如果有相同的请求到达时，就不需要再去发出请求，直接使用缓存即可。

域名解析为ip：

1、在浏览器中输入域名后，浏览器先在自己的缓存中查找是否有该域名对应的IP地址。
2、若没有，则操作系统会检查本地hosts文件是否有网址和IP地址的映射关系，若有则调用IP地址完成域名解析。
3、若hosts中没有这个域名的映射，则查找本地DNS解析器缓存
4、若还找不到，则会找本地DNS服务器，若本地DNS服务器区域没找到则会在本地DNS服务器的缓存中查找。
5、若本地DNS服务器区域和缓存都没查找到，则判断本地DNS服务器是否设置转发器。
6、若未使用转发模式，则把请求发送到根DNS服务器，根DNS服务器根据所发送来的域名来发送到下一级DNS服务器，递归的找到最终的IP地址。
7、若使用转发模式，则把请求发送至上一级DNS服务器，由上一级服务器进行解析并重复操作。
8、最终DNS服务器将查询结果返回给客户机，客户机根据所得到的IP信息访问目标主机完成域名解析。

当DNS寻址完成，把域名解析为IP地址之后。

建立连接

http基于tcp协议，tcp协议建立连接需要经历三次握手。
第一次握手：建立连接时，客户端发送syn包（syn=j）到服务器，并进入SYN_SENT状态，等待服务器确认；SYN：同步序列编号（Synchronize Sequence Numbers）。
第二次握手：服务器收到syn包，必须确认客户的SYN（ack=j+1），同时自己也发送一个SYN包（seq=k），即SYN+ACK包，此时服务器进入SYN_RECV状态。
第三次握手：客户端收到服务器的SYN+ACK包，向服务器发送确认包ACK(ack=k+1），此包发送完毕，客户端和服务器进入ESTABLISHED（TCP连接成功）状态，完成三次握手。

若只有两次握手只能确保客户端向服务器发送的消息，而由于服务器没有得到客户端的确认回应，所以服务器向客户端发送消息得不到保证。

和tcp同属传输层的另一个协议udp 协议不需要建立连接，udp发出包也不用确保是否得到有效响应，故udp协议不可靠，但传输效率更高。
故udp协议可被用于一下几个方面：
1、包总量较少的通信(譬如上面讲到的DNS)
2、对实时性要求性比较高的场景，比如直播、音视频通信。

服务器响应和返回

在客户端和服务器建立连接之后，可以互相传送数据，浏览器按照http协议格式将数据组装好向服务器发起请求，服务器收到请求并将处理结果响应给浏览器。

断开连接

四次挥手断开连接
第一次挥手：主动关闭方发送一个FIN，用来关闭主动方到被动关闭方的数据传送，也就是主动关闭方告诉被动关闭方：我已经不会再给你发数据了(当然，在fin包之前发送出去的数据，如果没有收到对应的ack确认报文，主动关闭方依然会重发这些数据)，但是，此时主动关闭方还可以接受数据。
第二次挥手：被动关闭方收到FIN后，可能还有数据没有传完，所以并不会马上向主动关闭方发送FIN，而是先发送一个确认序号ACK，意思是说“你发的断开连接请求我收到了，但是我现在还有数据没有发完，请稍等一下呗”。
第三次挥手：被动关闭方方在处理完数据报文后，便发送FIN报文给主动关闭方，也就是告诉对方，我的数据也发送完了，不会再给你发送数据了；发送完FIN报文后，被动方进入LAST_ACK阶段（超时等待）。
第四次挥手：主动关闭方收到FIN之后发送ACK给被动关闭方，确认序号为收到序号+1，至此，完成四次挥手。

补充HTTP与HTTPS的区别

HTTPS需要用到CA申请证书；
HTTP封装的信息是明文的，HTTPS封装的信息是加密的。
HTTP使用80端口，HTTPS使用443端口
默认的HTTP需要进行跳转，又被劫持的风险，HTTPS更加安全

管理事务处理

事务处理（transaction processing）可以用来维护数据库的完整性，它保证成批的MySQL操作要么完全执行，要么完全不执行。
若实际使用过程中，由于某种数据库故障，使得两条语句之间发生了中断，则会发生错误，为解决这种问题，使用事务处理机制，用来管理必须成批执行的 MySQL操作以保证数据库不包含不完整的操作。它们会作为整体执行，或完全不执行。若没有错误发生整组语句提交给数据库表，若发生错误，则进行回退，以恢复数据库到某个已知且安全的状态。

事务（ transaction ）指一组SQL语句；
回退（ rollback ）指撤销指定SQL语句的过程；
提交（ commit ）指将未存储的SQL语句结果写入数据库表；
保留点（ savepoint ）指事务处理中设置的临时占位符（place-holder），你可以对它发布回退（与回退整个事务处理不同）。

改善性能

(1) MySQL是用一系列的默认设置预先配置的，从这些设置开始通常是很好的。但过一段时间后你可能需要调整内存分配、缓冲区大小等。（为查看当前设置，可使用 SHOW VARIABLES; 和 SHOW STATUS; 。）
(2) MySQL一个多用户多线程的DBMS，换言之，它经常同时执行多个任务。如果这些任务中的某一个执行缓慢，则所有请求都会执行缓慢。如果你遇到显著的性能不良，可使用 SHOW PROCESSLIST 显示所有活动进程（以及它们的线程ID和执行时间）。你还可以用KILL 命令终结某个特定的进程（使用这个命令需要作为管理员登录）。
(3) 总是有不止一种方法编写同一条 SELECT 语句。应该试验联结、并、子查询等，找出最佳的方法。
(4) 使用 EXPLAIN 语句让MySQL解释它将如何执行一条 SELECT 语句。
(5) 一般来说，存储过程执行得比一条一条地执行其中的各条MySQL语句快。
(6) 应该总是使用正确的数据类型。
(7)决不要检索比需求还要多的数据。换言之，不要用 SELECT * （除非你真正需要每个列）。
(8)有的操作（包括 INSERT ）支持一个可选的 DELAYED 关键字，如果使用它，将把控制立即返回给调用程序，并且一旦有可能就实际执行该操作。
(9)在导入数据时，应该关闭自动提交。你可能还想删除索引（包括FULLTEXT 索引），然后在导入完成后再重建它们。
(10)必须索引数据库表以改善数据检索的性能。确定索引什么不是一 件微不足道的任务，需要分析使用的 SELECT 语句以找出重复的 WHERE 和 ORDER BY 子句。如果一个简单的 WHERE 子句返回结果所花 的时间太长，则可以断定其中使用的列（或几个列）就是需要索 引的对象。
(11)使用多条SELECT 语句和连接它们的 UNION 语句而不是一系列复杂的 OR 条件
(12)索引改善数据检索的性能，但损害数据插入、删除和更新的性能。 如果你有一些表，它们收集数据且不经常被搜索，则在有必要之 前不要索引它们。（索引可根据需要添加和删除。）
(13) LIKE 很慢。一般来说，最好是使用 FULLTEXT 而不是 LIKE 。

触发器按每个表每个事件每次地定义，每个表每个事件每次只允许一个触发器。因此，每个表最多支持6个触发器（每条 INSERT 、 UPDATE和 DELETE 的 BEFORE和AFTER）。单一触发器不能与多个事件或多个表关联，所以，若需要一个对 INSERT 和 UPDATE 操作执行的触发器，则应该定义两个触发器。

INSERT 触发器
在 INSERT 触发器代码内，可引用一个名为 NEW 的虚拟表，访问被插入的行；在 BEFORE INSERT 触发器中， NEW 中的值也可以被更新（允许更改被插入的值）；对于 AUTO_INCREMENT 列， NEW 在 INSERT 执行之前包含 0 ，在 INSERT执行之后包含新的自动生成值。

DELETE 触发器
在 DELETE 触发器代码内，可以引用一个名为 OLD 的虚拟表，访问被删除的行；OLD 中的值全都是只读的，不能更新。

UPDATE 触发器
在 UPDATE 触发器代码中，你可以引用一个名为 OLD 的虚拟表访问以前（ UPDATE 语句前）的值，引用一个名为 NEW 的虚拟表访问新更新的值；在 BEFORE UPDATE 触发器中， NEW 中的值可能也被更新（允许更改将要用于 UPDATE 语句中的值）； OLD 中的值全都是只读的，不能更新。

有时，一个完整的操作需要多条语句才能完成，例如：
1、为了处理订单，需要核对以保证库存中有相应的物品。
2、果库存有物品，这些物品需要预定以便不将它们再卖给别的人，并且要减少可用的物品数量以反映正确的库存量。
3、库存中没有的物品需要订购，这需要与供应商进行某种交互。

mysql命令行客户机的分隔符

可以临时更改命令行程序的语句分隔符。
